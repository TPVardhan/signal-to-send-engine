# signal-to-send email finder

A small FastAPI service that finds published contact emails on business websites. POST a website URL to `/find-email` and it fetches the homepage (realistic browser User-Agent, redirects followed), scans the HTML for `mailto:` links and plain-text emails, and — if nothing turns up — follows up to 3 links that look like contact/about/team pages and scans those too. Junk matches (image filenames, example.com placeholders, Wix/Sentry/GoDaddy system addresses) are filtered out, emails on the site's own domain are preferred, and robots.txt is respected: if the site disallows crawling you get `"email_status": "robots_disallowed"` and no further fetching happens. A `GET /` health check returns `{"status": "alive"}` for Render and n8n pings.

**Hybrid fetch strategy.** Every lookup tries httpx first (10-second timeout) because a plain HTTP fetch is fast and cheap and covers most sites. If the domain doesn't resolve or nothing answers the connection, the site is reported as `"site_unreachable"` immediately — a dead domain can't be browsed, so there is no Playwright attempt (and for lead scoring, a dead website is itself a distress signal). But if the server *answers* with a 4xx/5xx (anti-bot walls like Cloudflare 403s), or loads fine yet shows no email (some sites render contact info with JavaScript), the service retries with headless Chromium via Playwright: 20-second timeout, waits for network idle or at most 5 extra seconds after load, then re-runs the same email extraction on the rendered HTML. The response's `"method"` field tells you which client found the result (`"httpx"` or `"playwright"`, `null` if neither got through), and `"email_status"` is one of `published_on_site`, `no_email_published`, `site_unreachable`, `blocked`, or `robots_disallowed`.

**Run locally.** Create a virtualenv, `pip install -r requirements.txt`, then `playwright install chromium` (one-time browser download; without it the service still works, just without the browser fallback). Start the server with `uvicorn main:app --reload --port 8000` and test with `curl -X POST http://localhost:8000/find-email -H "Content-Type: application/json" -d "{\"website\": \"https://some-business.com\"}"`.

**Deploy on Render.** Create a Web Service from this repo with Build Command `./render_build.sh` (installs Python deps, then Chromium with its system libraries — commit the script with the executable bit set, `git update-index --chmod=+x render_build.sh`) and Start Command `uvicorn main:app --host 0.0.0.0 --port $PORT`. On the free tier the instance sleeps after inactivity and the first request after a sleep is slow; pinging `GET /` from n8n on a schedule keeps it awake. Chromium plus its dependencies is a large build — if the build cache misbehaves or `--with-deps` hits a permissions error on your plan, set the environment variable `PLAYWRIGHT_BROWSERS_PATH=/opt/render/project/.playwright` so the browser download lands inside the project directory and survives into the runtime container.
